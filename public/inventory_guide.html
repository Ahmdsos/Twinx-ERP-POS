1. المنتجات Products الصفحة الأهم

هذه الصفحة هي قلب النظام، تضم أكثر من 60 عموداً تغطي كل تفاصيل المنتج من الأبعاد والوزن حتى الربط المحاسبي الدقيق.

بيانات التعريف (Identity)
ID System Only
الرقم المرجعي الوحيد للمنتج في قاعدة البيانات.
قاعدة الاستخدام: لو بتعدل منتج موجود، لا تلمس هذا الرقم. لو بتضيف منتج جديد، سيب الخانة فاضية.
SKU Must Be Unique
كود تتبع الصنف (Stock Keeping Unit).
قاعدة الاستخدام: كود فريد لكل صنف. النظام يستخدمه لربط العمليات الحسابية ومنع التكرار.
Barcode Recommended
الباركود الدولي أو المحلي المطبوع على عبوة المنتج.
قاعدة الاستخدام: ادخله عشان تقدر تبيع وتجرد بسرعة باستخدام الماسح الضوئي (Scanner).
Name Required
اسم المنتج كما سيظهر للعميل وفي الفاتورة.
قاعدة الاستخدام: اكتب الاسم بدقة (مثلاً: موبايل سامسونج S24 الترا - أسود).
Description Detailed
شرح إضافي للمنتج أو مواصفاته التقنية.
يظهر في تقارير الجرد وفي صفحة تفاصيل المنتج.
Type Default: goods
نوع الصنف (Goods أو Service).
قاعدة الاستخدام: المنتجات الملموسة اكتب goods، والخدمات اكتب service.
الربط والهيكلة (Associations)
Category ID Required
رقم القسم الذي ينتمي إليه المنتج.
قاعدة الاستخدام: اذهب لصفحة Categories، ابحث عن رقم (ID) القسم المناسب وضعه هنا.
Brand ID Optional
رقم الماركة أو الشركة المصنعة.
قاعدة الاستخدام: هات الـ ID من صفحة Brands. لو ملوش ماركة سيبه فاضي.
Unit ID Required
وحدة البيع الافتراضية (قطعة، كيلو، إلخ).
قاعدة الاستخدام: هات الـ ID من صفحة Units. تأكد إنها الوحدة الصغرى للجرد.
Purchase Unit ID Defaults to Unit
الوحدة اللي بتشتري بيها من المورد.
قاعدة الاستخدام: لو بتشتري بالكرتونة وتبيع بالقطعة، حط ID الكرتونة هنا.

سطوة الكود (JSON Power Mode) 💻

هذا هو الطريق المختصر للتحكم الكامل. ملف JSON واحد يمثل الـ 5 صفحات معاً، جاهز للتعديل البرمجي أو عبر أي Text Editor.

1. تصدير الحقيقة

بضغطة واحدة من واجهة النظام أو عبر الكوماند:

php artisan twinx:inventory-export
2. التعديل الشامل

افتح ملف twinx_inventory_export.json واستخدم سحر الـ Search & Replace.

3. إعادة الاستيراد

ارفع الملف من شاشة المنتجات أو استخدم الكوماند:

php artisan twinx:inventory-import

لماذا الـ JSON هو الخيار الأفضل؟

✅ هيكل موحد (Single File):
لا حاجة للتنقل بين 5 ملفات أو شيتات. كل شيء في مكان واحد.
✅ دعم الصور (Image Links):
ملف الـ JSON يدعم ربط الصور بمساراتها الأصلية بسهولة.
✅ ذكاء الجرد (Target Socking):
النظام بيقارن الكمية في الملف بالواقع وبيعمل عمليات Adjustment أوتوماتيك.
✅ أمان المعاملات (Atomic):
لو حصلت غلطة في منتج واحد، النظام بيرفض الملف كله عشان يحمي بياناتك.

التحكم البرمجي الكامل في الأقسام، الماركات، الوحدات، المخازن، والمنتجات.

تحميل المرجع الشامل JSON
دليل ذكاء الجيسون (JSON Syntax Mastery)

قبل البدء، يجب أن تفهم أن ملف الـ JSON هو "كود برمجي" حساس جداً. أي خطأ في حرف أو علامة سيؤدي لفشل الاستيراد بالكامل.

الأقواس (Braces/Brackets)
تستخدم {} لتعريف "كائن" (Object) مثل بيانات منتج واحد، بينما [] لتعريف "مصفوفة" (Array) تضم عدة منتجات.
أنواع البيانات (Data Types)
- String: نص بين علامتي تنصيص "مثل هذا".
- Integer/Float: أرقام بدون علامات تنصيص 1500.50.
- Boolean: قيم منطقية true أو false.
الفواصل اللعينة (Trailing Commas)
لا تضع فاصلة , بعد آخر عنصر في الكائن أو المصفوفة. هذا أشهر سبب لفشل الكود.
التحقق (Validation)
دائماً اختبر الكود الخاص بك في موقع مثل JSONLint قبل المحاولة في توينكس.
1. الأقسام (Categories Mastery)

هذا الجدول يشرح كل مفتاح في مصفوفة categories. الأقسام هي حجر الأساس لتنظيم المخزن.

idInteger
المعرف الفريد للقسم.
الأهمية: هو الرقم الذي نضعه في حقل category_id داخل المنتجات لربطها بهذا القسم.
nameString
الاسم المعروض (مثل: "موبايلات"، "إكسسوارات").
slugString
الرابط اللطيف (Clean URL).

parent_idInteger / Null
الربط الشجري (Hierarchy).
لو هذا "قسم فرعي"، نضع هنا ID "القسم الأب". لو قسم رئيسي، نضعه null.
descriptionString
وصف تفصيلي للقسم ومهامه.
sort_orderInteger
ترتيب العرض (0 هو الأعلى).
is_activeBoolean
حالة القسم (نشط/معطل).
2. الماركات (Brands Mastery)

تعريف الشركات المصنعة لربط الأصناف بأصولها العالمية ومنشئها. البراند يساعد في فلترة التقارير وبناء صفحات خاصة للعملاء.

idInteger
المعرف الفريد للبراند (Primary Key).
قاعدة الربط: يستخدم في حقل brand_id داخل مصفوفة المنتجات لربط المنتج بماركته.
nameString
اسم الماركة الرسمي (مثل: Apple, Samsung, Dell).
معلومة: الاسم يظهر بجانب اسم المنتج في الفاتورة والمتجر.
websiteURL String
الموقع الإلكتروني الرسمي للشركة المصنعة.
descriptionText String
نبذة عن تاريخ الماركة أو تخصصها.
is_activeBoolean
حالة الماركة في النظام (متاحة للاستخدام أم لا).
3. وحدات القياس (Units Mastery)

إدارة الوحدات الأساسية والتحويلات البرمجية الدقيقة.

idInteger
المعرف الفريد للوحدة.
nameString
الاسم الكامل (مثل: "قطعة"، "كرتونة").
abbreviationString
الاختصار (مثل: pcs, box).
is_baseBoolean
هل هي وحدة أساسية؟
الوحدة الأساسية هي التي لا تشتق من غيرها (مثل "القطعة").
base_unit_idInteger / Null
الوحدة المرجعية.
لو الوحدة "كرتونة"، بنحط هنا ID "القطعة" عشان نحسب التحويل.
conversion_factorFloat
معامل التحويل.
لو الكرتونة فيها 12 قطعة، المعامل بيكون 12.0. لو وحدة أساسية بيكون 1.0.
4. المخازن (Warehouses Mastery)

تحديد القنوات اللوجستية وتوزيع المخزون الفيزيائي. المخازن هي أوعية البيانات الكمية.

idInteger
المعرف الفريد للمخزن في السيستم.

codeString
كود المخزن المختصر (مثل: WH-MAIN, STORE-01).
احترافي: الكود يساعد الموظفين في التمييز السريع بين المواقع عند التحويل المخزني.
nameString
الاسم الكامل للمخزن المعترف به إدارياً.
addressString
العنوان الجغرافي التفصيلي للمخزن.
phone / emailString
بيانات التواصل المباشر مع أمين المخزن.
manager_idInteger
ID المستخدم المسؤول عن إدارة هذا الموقع.
is_defaultBoolean
هل هذا هو "المخزن الافتراضي" لعمليات البيع؟
النظام يسحب الكميات آلياً من هذا المخزن ما لم يحدد الكاشير غير ذلك.
is_activeBoolean
حالة المخزن (متاح للعمليات أم معطل للصيانة/الجرد).

الربط المعقد (Atomic Relational Objects)

هذه الحقول هي أوعية (Objects) وليست مجرد نصوص. دقتها تحدد سلامة حركة المخزن.

stocks Object {WH_ID: Qty}
خريطة الحقيقة المكانية للمخزون.
طريقة الكتابة:
"stocks": { "1": 150, "2": 45 }
المعنى: المخزن رقم 1 يحتوي على 150 قطعة.
images_list Array of Objects
معرض صور المنتج الرقمي.
تنسيق الكائن:
{ "path": "p/img.jpg", "is_primary": true, "sort_order": 0 }

قواعد سطوة الكود (Pro Engineering Standards)
السرعة الذرية (Atomic Speed): ملف الـ JSON يتم معالجته في ذاكرة النظام (Memory-Speed) وهو أسرع ב- 10 مرات من الإكسل. ينصح به لقواعد البيانات التي تتخطى الـ 100 ألف صنف.
السلامة المعيارية (Atomic Integrity): عند استيراد JSON، النظام يطبق مبدأ الـ Transaction. لو مسمار واحد في الملف غلط، العملية كلها "بتفشل" عشان نحمي داتا شركتك من التلوث (Zero Data Pollution).
حقائق برمجية عن JSON وتوأم اكس (The Science of Twinx JSON)

1. لماذا لا نستخدم Excel فقط؟

لأن الإكسل لا يدعم "العلاقات المتشعبة". في JSON يمكننا وضع مخازن وصور ومواصفات داخل صنف واحد (One-to-Many) بشكل هرمي، وهو ما يستحيل فعله في ملف جدول مسطح بدون تكرار البيانات.

2. ما معنى تهرّب الرموز (Escaping)؟

إذا كان اسم المنتج يحتوي على علامة تنصيص "، يجب أن تسبقها بشرطة مائلة \" لكي لا يظن المفسر أن النص قد انتهى. مثال: "TV 55\" Inch".

3. الترتيب لا يهم!

في JSON، ترتيب المفاتيح لا يؤثر على الصحة. يمكنك وضع sku في الأول أو في الآخر، النظام سيتعرف عليه طالما الاسم (Key) مكتوباً بدقة.

4. ترميز اللغة (UTF-8)

دائماً احفظ ملفاتك بتنسيق UTF-8 لضمان ظهور الأحرف العربية بشكل صحيح وعدم تحولها لرموز غريبة عند الرفع.
